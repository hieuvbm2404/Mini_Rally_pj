# P2-IS-03 - Backlog Iteration Assignment

## Scope for this checkpoint

This execution intentionally tests only the dependency requested by BA:

1. Assign an existing Story/Defect to an Iteration from Backlog.
2. Confirm the assignment persists on the same Work Item ID.
3. Confirm Work Item Detail reflects the same Iteration.
4. Confirm the same ID is visible in the selected Iteration on Iteration Status.
5. Confirm assignment does not automatically change Iteration status to Committed.

Task Dashboard, Task roll-up, Iteration totals, Add Item and auto-Accept remain outside this checkpoint until BA confirms the assignment baseline.

## Confirmed business baseline used for comparison

- Backlog remains the source list for Story/Defect; assigning an Iteration updates the existing Work Item rather than creating a copy.
- One Work Item may be Unscheduled or assigned to one active Iteration.
- Backlog, Work Item Detail and Iteration Status must resolve the same Work Item and Iteration identities.
- Assigning a Work Item does not auto-change the Iteration from Planning to Committed.
- Committed does not lock scope.
- All Teams is an allowed Phase 2 context.

## Mockup execution

- Source Work Item: `US-4822 / MOCK CWD AUDIT 20260719 2258`.

- Iteration Status for `Sprint 24.5` showed the same `US-4822` exactly once.
- Iteration remained `Planning`; assignment did not auto-Commit.
- Task metrics shown on the mockup were not assessed in this checkpoint.

Result: assignment identity and cross-screen behavior match the approved mockup baseline.

## Observed gaps pending BA confirmation

### GAP-P2-IS-001 - Iteration inline assignment UI synchronization

The update persists, but Backlog immediately renders the old value until reload. Dev should update the row from the canonical mutation response or invalidate/refetch the correct scoped Backlog query. This is the same save/cache contract already observed for inline Title editing.

### GAP-P2-IS-002 - Iteration Status first-load false empty state

On the first direct visit in `NXP / All Teams`, Iteration Status displayed a final-looking no-Iterations empty state. After Timeboxes loaded and the user returned, the same screen correctly exposed all three Iterations and the assigned Work Item. Iteration Status should initialize its Iteration query independently, show Loading while unresolved, and render the empty state only after the scoped query settles successfully with zero records.
